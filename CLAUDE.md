# CLAUDE.md

Guidance for working in this repo. Read this before changing build config, SDK
component selection, or the target device.

## What this is

A self-contained, IDE-free build template for NXP MCUs built on the MCUXpresso
SDK (west workspace), CMake and Ninja. `exec.bat <script> [args]` runs
`scripts/<script>.py` inside a bootstrapped Python venv. The bootstrap
(`env/env.py`) creates the venv, does the west SDK checkout into `vendor/`, and
downloads a portable ARM GCC / CMake / Ninja into `env/tools/`. Windows-only for
now. See `README.md` for the user-facing version of all of this.

It is a **device-only** SDK build: there is no SDK board layer. Pin mux, clock
and peripheral setup live in `src/board/`, generated by MCUXpresso Config Tools
from `src/<board>.mex`. Currently one project: `src/` (MCXA266 / FRDM-MCXA266,
blue LED blink on P3_21).

## Layout that matters

- `config.yaml` — the single place for device/board settings (`device`,
  `linker_script_path`, `linkserver_device`). `scripts/build.py` forwards these
  to CMake as `-D` vars.
- `src/prj.conf` — Kconfig component selection (`CONFIG_MCUX_COMPONENT_<id>=y`).
  Decides which SDK drivers/components compile in.
- `src/Kconfig` — custom Kconfig root (sources devices/drivers/components only;
  deliberately skips the SDK's board/examples Kconfig this repo doesn't fetch).
- `src/CMakeLists.txt` — globs all `src/**/*.{c,h,cpp,hpp}`, so **everything
  under `src/` (including all of `board/`) is always compiled.** A driver used
  by any board file must be enabled in `prj.conf`, even if `main.cpp` never
  touches it.
- `src/board/` — Config Tools output. Generated, NXP-copyrighted (BSD-3-Clause).
- `env/env.py` — bootstrap; the `west update` project list lives here.

## Building / verifying

`exec.bat build` (configure + build, `MinSizeRel`), `--clean` to wipe `build/`,
`--jobs N` to limit. Other scripts: `exec.bat elfinspect`, `exec.bat flash`.
A build is the real check after any `prj.conf`/device change — the toolchain and
SDK must actually fetch and compile.

## Changing prj.conf (enabling a "new" SDK part / driver / component)

`prj.conf` symbols are `CONFIG_MCUX_COMPONENT_<component id>=y`. The id matches
the SDK Kconfig (e.g. `driver.lpuart`, `driver.lpi2c`, `component.serial_manager`,
`utility.debug_console`).

When the user wants a new SDK feature (say I2C, a UART, the debug console):

1. Find the component id in the fetched SDK Kconfig under `vendor/mcuxsdk/`
   (`drivers/<x>/Kconfig`, `components/<x>/Kconfig`, `devices/.../Kconfig`).
   Grep for `config MCUX_COMPONENT_` to see exact ids and their `depends on`.
2. Add `CONFIG_MCUX_COMPONENT_<id>=y` to `prj.conf`. Add its dependencies too —
   Kconfig here does not always auto-select them, and a missing dep usually
   shows up as a link error or a missing `fsl_<x>.h`.
3. Keep `prj.conf` **minimal and justified**: each line should correspond to
   code that actually compiles (something in `src/` includes its `fsl_<x>.h`).
   Because CMake globs all of `board/`, a driver pulled in only by a board file
   (e.g. `driver.mcx_spc`, used by `board/clock_config.c` via `fsl_spc.h`) still
   has to be present. The existing file documents each entry this way — match
   that style.
4. Re-run `exec.bat build`.

To *remove* a feature, delete its `prj.conf` lines, but first confirm no file
under `src/` (especially `board/`) still references it, or the build breaks.

## Changing to a different MCU / board

The device/board lives in **`config.yaml`** plus the Config Tools artifacts.
Full procedure:

1. **`config.yaml`** — set:
   - `device` — the MCUXpresso SDK device name (CMake `device`, e.g. `MCXA266`).
   - `linker_script_path` — relative to repo root; usually
     `vendor/mcuxsdk/devices/<...>/gcc/<DEVICE>_flash.ld`, or a custom script
     committed to the repo.
   - `linkserver_device` — `<device>:<board>` for flashing
     (`LinkServer devices` lists valid names).
2. **Make sure the device family is fetched** by west. The `west update` project
   list is in `env/env.py` → `_ensure_sdk()` (currently `core`, `CMSIS`,
   `mcu-sdk-components`, `mcux-devices-rt`, `mcux-devices-mcx`). If the new part
   is a different family (e.g. an i.MX RT vs. MCX), add the matching
   `mcux-devices-*` project there. Because the bootstrap only inits the west
   workspace when it's missing, **delete `vendor/.west` and `vendor/mcuxsdk`
   (both gitignored) to force a re-sync**, then re-run `exec.bat build`. (Same
   forced-resync caveat as changing the SDK release — see README "Changing the
   SDK release".)
3. **Regenerate the board config with MCUXpresso Config Tools** (this is the
   part that cannot be scripted — it needs the GUI tool):
   - Create/open a `.mex` for the new part/board in Config Tools, or start from
     NXP's stock board `.mex` for that board.
   - Configure pins / clocks / peripherals as needed.
   - Generate code into `src/board/` (`pin_mux`, `clock_config`, `peripherals`,
     `.c` + `.h`). These overwrite the existing board files.
   - Commit the new `.mex` as `src/<BOARD>.mex` and remove the old one.
   - The generated files carry NXP `Copyright` / `BSD-3-Clause` headers — leave
     them intact (see Licensing below).
4. **Reconcile `prj.conf`** — the new board's `clock_config.c` / `pin_mux.c` /
   `peripherals.c` may use different drivers (e.g. a different SPC, a different
   UART instance). Update the component selection to match what the new board
   files include, following the "Changing prj.conf" rules above.
5. **`main.cpp`** — board-specific symbols (e.g. the LED `BOARD_INITLEDSPINS_*`
   macros, the `BOARD_BootClock*` choice) come from the regenerated headers;
   update `main.cpp` to the new pins/clock entry point.
6. Build and, if hardware is present, flash.

Note: changing the *SDK release* (the `vendor/mcuxsdk-manifests` submodule) is a
separate operation documented in `README.md` → "Changing the SDK release". The
trigger for that is usually a Config Tools version bump.

## Conventions

- Application code is C++20, `-fno-exceptions -fno-rtti` (see `CMakeLists.txt`).
  The SDK stays C, links newlib-nano.
- Prefer **SDK functions over hand-rolled helpers** (e.g. `GPIO_PortToggle`,
  `SDK_DelayAtLeastUs` rather than custom toggle/delay loops) — the maintainer
  has stated this preference explicitly.
- Do not hand-edit `src/board/*` or `.mex`-generated files; they are overwritten
  by Config Tools. Configure in the tool and regenerate.
